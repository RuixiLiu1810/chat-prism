# Agent CLI 全功能 TUI 设计（S3）

## 1. 背景

当前 `agent-cli` 已具备可用的 REPL 与工具执行链路，但整体交互仍偏“日志输出模式”，在会话连续性、信息层级与可发现性上明显弱于 Claude Code 风格体验。

已确认方向不是继续局部修补，而是采用 `S3`：建设全功能 TUI 框架，在保持现有运行能力的前提下，重建主界面与交互语法。

## 2. 目标

1. 提供默认常驻的全屏命令行主界面（非临时打印头部）。
2. 建立稳定的信息层级：`用户输入 / 助手回复 / 系统语义回执`。
3. 默认显示语义结果，支持按需展开技术细节。
4. 修复 `suspended` 语义：在同一轮会话内等待审批与继续执行，不制造“重开一轮”的视觉错觉。
5. 在不改动 `agent-core` 协议前提下完成 UI 层升级。

## 3. 范围与非目标

### 3.1 In Scope

1. L1 三段式布局：轻量头部 + 对话区 + 单行输入区。
2. I1 输入模式：单行输入（CLI-first）。
3. 语义结果默认显示，细节可展开。
4. `suspended/approval` 同轮恢复。
5. 保留 `jsonl` 输出路径与现有非 TUI 运行能力（兼容调试与自动化）。

### 3.2 Out of Scope

1. 双栏工作台（L2）与极简流（L3）并行实现。
2. 多行输入框（I3）与富文本输入。
3. 鼠标驱动的复杂交互。
4. `agent-core` 事件协议重构。

## 4. 方案选型

## 4.1 备选方案

1. `S1` 仅改输出渲染层：改动最小，但后续扩展受限。
2. `S2` 增加轻量 ViewModel 层：平衡型，但目标仍偏“增强 REPL”。
3. `S3` 全功能 TUI 框架：投入最大，但最符合“对标 Claude Code CLI 手感”的目标。

## 4.2 结论

采用 `S3`。

原因：
1. 用户目标是“做得好”，不是“勉强可用”。
2. 需要从根本上统一布局、语义与状态机，而非继续叠加日志样式补丁。
3. 后续命令体系 2.0、可发现性与上下文感更依赖稳定的 TUI 框架。

## 5. 架构设计

## 5.1 模块划分（agent-cli 内）

1. `tui/shell.rs`
   - 管理全屏生命周期与 `L1` 布局绘制。
2. `tui/view_model.rs`
   - 将底层事件规约为时间线语义节点。
3. `tui/event_bridge.rs`
   - 承接 `EventSink` 事件并做语义映射。
4. `tui/input.rs`
   - 单行输入、命令路由、焦点与键位处理。
5. `tui/detail.rs`
   - 语义节点的展开/折叠细节视图。

## 5.2 与现有模块边界

1. 复用 `turn_runner`、`tool_executor`、`runtime_state` 既有能力。
2. 不改 `agent-core` 的事件协议与 provider 逻辑。
3. `output=jsonl` 保持原有行为。

## 6. 交互模型与命令面

## 6.1 L1 布局

1. 顶部轻量头部字段：
   - `provider/model | project_path | git_branch(+dirty) | session_id | output_mode`
2. 中部对话区：
   - `›` 用户输入
   - `●` 助手回复
   - `└` 系统语义回执
3. 底部输入区：
   - I1 单行输入：`> input...`

## 6.2 默认显示策略

1. 默认隐藏技术噪声事件（如内部阶段标签）。
2. 默认展示语义结果（对用户可行动的信息）。
3. 每条 `└` 节点可展开技术细节（阶段、参数摘要、错误上下文）。

## 6.3 键位与操作（MVP）

1. `Enter`：提交输入 / 在时间线焦点上展开或折叠。
2. `Up/Down`：输入历史。
3. `Tab`：slash 命令补全。
4. `Ctrl+J / Ctrl+K`：时间线焦点移动。
5. `Esc`：返回输入框。
6. `Ctrl+L`：清屏并重绘，不丢会话。

## 7. 数据流与状态机

## 7.1 启动流

1. 解析参数与配置。
2. 进入 `TuiShell::run()`。
3. 初始化 `ViewModel`、`EventBridge`、`InputController`。
4. 立即绘制首帧界面（不等待首轮请求）。

## 7.2 单轮执行流

1. 用户提交后写入 `›` 节点。
2. 触发 turn 执行。
3. `EventBridge` 接收底层事件并生成：
   - `●` 助手文本节点
   - `└` 语义结果节点
4. 同步保存 detail 缓冲供展开查看。

## 7.3 审批与挂起流（关键）

1. 收到 `approvalRequired` 或 `suspended`：
   - 追加 `└ Waiting for approval ...`
   - 状态转为 `WaitingApproval`
   - 不重置会话、不重绘启动头
2. 用户执行 `/approve ...` 后：
   - 更新审批状态
   - 在同轮继续执行并追加后续语义节点

## 8. 错误处理

1. 配置错误：语义节点提示 + 下一步操作建议（如 `/config`）。
2. Provider/网络错误：主视图给简明失败语义，detail 保留原始错误。
3. 工具错误：主视图语义化，detail 保留 stderr/exit code。
4. TUI 渲染错误：降级到现有 human 文本输出，保证可用性。
5. 用户中断：追加 `Interrupted by user` 语义节点并恢复输入焦点。
6. `suspended` 归类为可恢复状态，不归类为失败终止。

## 9. 测试面

## 9.1 单元测试

1. `ViewModel` 事件规约正确性。
2. `suspended/approval` 状态迁移正确性。
3. detail 展开/折叠行为。
4. 输入控制与命令路由。

## 9.2 组件与渲染测试

1. 窄终端/宽终端下 `L1` 不破版。
2. 局部刷新策略不闪烁、不错位。

## 9.3 集成测试

1. 多轮对话 + 工具调用 + 审批恢复同轮连续。
2. `jsonl` 输出行为不回归。
3. provider/model 变更后头部字段一致。

## 10. 成功标准（Definition of Done）

1. 启动即进入常驻全屏 `L1` 界面。
2. 输入区采用 `I1` 单行模式并稳定可用。
3. 默认只显示语义结果，细节可展开查看。
4. `suspended` 不产生“新会话”视觉效果，审批后同轮继续。
5. 现有 `jsonl` 与非 TUI 路径可继续使用。
6. 关键测试通过且无核心交互回归。

## 11. 实施顺序建议

1. 先落地 `TuiShell + L1` 空骨架与输入焦点。
2. 接入 `EventBridge + ViewModel`，打通语义时间线。
3. 接入 detail 展开能力。
4. 完成 `suspended/approval` 同轮状态机。
5. 补齐测试并做回归验收。
